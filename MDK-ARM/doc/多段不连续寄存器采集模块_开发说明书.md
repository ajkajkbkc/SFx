# 多段不连续寄存器采集模块 — 开发说明书

## 一、概述

### 1.1 问题背景

原有 Modbus RTU 采集系统的设计前提是：**每个从站设备的所有寄存器地址是连续的**，只需发送一条 Modbus 0x03 读保持寄存器指令即可采集全部数据。当遇到寄存器地址不连续（分成多段）的从站设备时，原有架构无法支持。

### 1.2 解决方案

引入 **"寄存器段映射表"** 机制，将设备类型的控制点按寄存器地址区间拆分为多个 **"段（Segment）"**，每个段对应一段连续寄存器区间。采集时对每个段分别发送 Modbus 读指令并等待响应，最后将各段数据合并到统一控制点数组中。

### 1.3 核心设计思路

```
设备类型 → 寄存器映射表 → Segment 0: {起始地址, 寄存器数, 控制点偏移, 控制点数}
                        → Segment 1: {起始地址, 寄存器数, 控制点偏移, 控制点数}
                        → Segment N: {起始地址, 寄存器数, 控制点偏移, 控制点数}
```

- 现有设备（单段连续寄存器）→ 定义 1 个 Segment，行为与原有代码**完全一致**
- 新增多段设备 → 定义 N 个 Segment，框架自动按段依次轮询

## 二、数据结构定义

### 2.1 `reg_segment_def_t` — 寄存器段定义

```c
/**
  * @brief  寄存器段定义结构体
  *         用于描述一个从站设备的一段连续寄存器区间
  */
typedef struct
{
    uint16_t StartAddr;       /* Modbus 寄存器起始地址 */
    uint16_t RegCount;        /* 本段读取的寄存器数量 */
    uint8_t  CtrlP_StartOff;  /* 本段对应到该设备控制点数组的起始偏移(从0开始) */
    uint8_t  CtrlP_Count;     /* 本段包含的控制点个数 */
} reg_segment_def_t;
```

**字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `StartAddr` | `uint16_t` | Modbus 读指令中的寄存器起始地址（对应报文第2~3字节） |
| `RegCount` | `uint16_t` | 一次读取的寄存器个数（对应报文第4~5字节） |
| `CtrlP_StartOff` | `uint8_t` | 该段数据映射到设备控制点数组的起始索引（从0开始） |
| `CtrlP_Count` | `uint8_t` | 该段包含的控制点个数（通常 ≤ RegCount） |

### 2.2 `device_reg_map_t` — 设备寄存器映射表

```c
/**
  * @brief  设备寄存器映射表结构体
  *         将设备类型与其寄存器段表关联起来
  */
typedef struct
{
    uint8_t  SegCount;                    /* 寄存器段数(1=单段连续, 2+=多段不连续) */
    const reg_segment_def_t *pSegments;   /* 各段定义数组指针 */
} device_reg_map_t;
```

**字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `SegCount` | `uint8_t` | 寄存器段的数量。1 = 单段连续（现有设备）；>1 = 多段不连续 |
| `pSegments` | 指针 | 指向 `reg_segment_def_t` 数组的指针 |

### 2.3 全局变量 `gRtu_SegmentIdx`

```c
__IO uint8_t gRtu_SegmentIdx;  /* 当前正在采集的寄存器段索引(用于多段不连续寄存器设备, 从0开始) */
```

用于在采集循环和 UART 响应处理之间传递当前正在处理的段索引。

/* --- 单段设备定义 --- */

## 四、核心函数说明

### 4.1 `GetDeviceRegMap()` — 获取设备寄存器映射表

```c
const device_reg_map_t* GetDeviceRegMap(uint8_t RtuType);
```

- **功能**：根据设备类型枚举值，返回对应的寄存器映射表指针
- **参数**：`RtuType` — 设备类型（TYPE_DES3 ~ TYPE_G2A3 等）
- **返回**：指向 `device_reg_map_t` 的指针；无效类型返回 NULL

### 4.2 `SendSegmentReadCmd()` — 按段发送 Modbus 读指令

```c
static void SendSegmentReadCmd(uint8_t RtuId, const reg_segment_def_t *pSegDef);
```

- **功能**：根据段定义构造一条 Modbus 0x03 读保持寄存器指令并发送
- **参数**：
  - `RtuId` — 从站 Modbus 地址
  - `pSegDef` — 指向该段定义的指针

**指令格式**：
```
[RtuId] [0x03] [StartAddr高8位] [StartAddr低8位] [RegCount高8位] [RegCount低8位] [CRC16]
```

### 4.3 `Handle_RtuType_SegmentSend()` — 段发送对外接口

```c
void Handle_RtuType_SegmentSend(uint8_t RtuId, uint8_t RtuType, uint8_t SegIdx);
```

- **功能**：根据设备类型和段索引，查找段定义并发送对应的读取指令
- **参数**：
  - `RtuId` — 从站 Modbus 地址
  - `RtuType` — 设备类型
  - `SegIdx` — 段索引（0 ~ SegCount-1）

### 4.4 `Handle_MultiSeg_Response()` — 多段设备响应解析

```c
bool Handle_MultiSeg_Response(uint8_t *pBuf, uint16_t len, uint8_t RtuType, uint8_t SegIdx);
```

- **功能**：解析多段寄存器设备单段的响应数据，按**线性映射**赋值到控制点数组
- **参数**：
  - `pBuf` — 响应数据缓冲区（含 Modbus 帧头）
  - `len` — 数据长度
  - `RtuType` — 设备类型
  - `SegIdx` — 当前段索引
- **返回**：true = 解析成功，false = 解析失败

**解析规则**：
```
段内第 j 个寄存器（响应偏移 3+2*j） → CtrlP_StartOff + j 号控制点
```

Modbus 响应帧格式：
```
[0:站号] [1:功能码] [2:字节数] [3:4:数据0] [5:6:数据1] ... [CRC]
```

### 4.5 `is_RtuReturn_protocol()` — 修改后的响应入口

在原有函数开头增加多段设备检测分支：

```c
/* 多段设备检测 */
const device_reg_map_t *pMap = GetDeviceRegMap(gRtu_InfoPacket.Rtu_Info[gRtu_Idx].RtuType);
if(pMap != NULL && pMap->SegCount > 1)
{
    /* 调用分段解析函数，仅处理当前段(gRtu_SegmentIdx)的数据 */
    if(Handle_MultiSeg_Response(pBuf, len,
        gRtu_InfoPacket.Rtu_Info[gRtu_Idx].RtuType, gRtu_SegmentIdx) == true)
    {
        xSemaphoreGive(gGetOneOK_BinSem);  // 释放信号量
        return true;
    }
    return false;
}
/* 单段设备继续原有处理逻辑... */
```

### 4.6 `GetAllRtuValue()` — 修改后的采集循环

增加段级嵌套循环：

```c
for(gRtu_Idx = 0, gCtrlP_Idx = 0; gRtu_Idx < gRtu_Num; gRtu_Idx++)
{
    uint8_t rtuType = gRtu_InfoPacket.Rtu_Info[gRtu_Idx].RtuType;
    uint8_t rtuIdNum = gRtu_InfoPacket.Rtu_Info[gRtu_Idx].RtuIdNum;

    const device_reg_map_t *pMap = GetDeviceRegMap(rtuType);
    uint8_t segCount = (pMap != NULL) ? pMap->SegCount : 1;
    uint16_t devCtrlP_Start = gCtrlP_Idx;
    BaseType_t allSegOK = pdTRUE;

    /* === 段级轮询 === */
    for(gRtu_SegmentIdx = 0; gRtu_SegmentIdx < segCount; gRtu_SegmentIdx++)
    {
        if(pMap != NULL && pMap->SegCount > 1)
            Handle_RtuType_SegmentSend(rtuIdNum, rtuType, gRtu_SegmentIdx);
        else
            Handle_RtuType_ToSend(rtuIdNum, rtuType);

        if(xSemaphoreTake(gGetOneOK_BinSem, timeout))
            { /* 成功 */ }
        else
            { allSegOK = pdFALSE; break; }
    }

    /* 更新设备状态 */
    gCtrlP_Idx = devCtrlP_Start + CtrlPNum_Table[rtuType];
}
```

## 五、数据流

### 5.1 单段设备（原有路径，无变化）

```
GetAllRtuValue()
  → Handle_RtuType_ToSend()          // 发送1条Modbus指令
  → wait semaphore
  → is_RtuReturn_protocol()           // 解析全部控制点 + 报警判断
  → xSemaphoreGive()                  // 释放信号量
```

### 5.2 多段设备（新的路径）

```
GetAllRtuValue()
  → for 每个段 (gRtu_SegmentIdx = 0..SegCount-1):
      → Handle_RtuType_SegmentSend()  // 发送该段的Modbus指令
      → wait semaphore
      → is_RtuReturn_protocol()
          → Handle_MultiSeg_Response() // 仅解析当前段的控制点
          → xSemaphoreGive()           // 释放信号量
  → 累加 gCtrlP_Idx
```

**关键点**：每个段独立发送指令 → 等待响应 → 解析数据 → 释放信号量，形成完整的"发-等-收"周期。

## 六、新增设备适配步骤

当需要接入一个新的多段不连续寄存器从站设备时，按以下步骤操作：

### 步骤 1：定义设备类型枚举

在 `app_collect.h` 中新增 `TYPE_xxx` 枚举值和 `CtrlPNum_Table` 条目：

```c
#define  TYPE_NEW     14    // 新设备类型
#define  MAX_PRODEUCT_TYPE  15  // 更新总数
```

```c
const uint8_t CtrlPNum_Table[MAX_PRODEUCT_TYPE] = {
    // ...原有...
    5  // TYPE_NEW 有5个控制点
};
```

### 步骤 2：定义寄存器段表

在 `app_collect.c` 的段定义区域添加：

```c
static const reg_segment_def_t SegDef_NewDevice[] = {
    {0x0010, 2, 0, 2},    /* 段0: 地址0x0010, 2个寄存器 → 控制点0~1 */
    {0x0100, 3, 2, 3},    /* 段1: 地址0x0100, 3个寄存器 → 控制点2~4 */
};
```

### 步骤 3：注册到设备映射表

在 `s_DeviceRegMap[]` 中添加条目：

```c
/* TYPE_NEW */ {2, SegDef_NewDevice},  // 2段不连续寄存器
```

### 步骤 4：更新各回调函数

按照注释要求更新以下函数以支持新类型：
- `gatewayparam_init()` — ID 识别
- `Handle_RtuCtrlP_AlmLimType()` — 报警门限类型
- `Handle_RtuCtrlP_PyldType()` — 载荷数据类型
- `Handle_RtuCtrlP_CtrlPTypeStr()` — 控制点类型字符串
- `Handle_FindFlkRtu_RecvData()` — 自动搜索识别
- `is_RtuReturn_protocol()` — 响应解析（多段设备已由 Handle_MultiSeg_Response 处理）

## 八、风险与注意事项

1. **超时处理**：多段设备某段超时时，跳过该设备所有剩余段，避免阻塞总线
2. **段间延迟**：每段之间保留 `getRtuInterval` 的延迟，给从站响应时间
3. **控制点映射**：段内寄存器按线性顺序映射到控制点（寄存器0 → CtrlP_StartOff + 0），不支持交错映射
4. **响应长度校验**：`Handle_MultiSeg_Response()` 会对响应数据长度做校验，防止越界访问
5. **信号量匹配**：每个段发送后等待一次信号量，确保"发-等-收"一对一匹配
